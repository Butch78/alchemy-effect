`v2.0.0-beta.43` ships two things worth calling out. The first is
**PlanetScale** — both MySQL and the new Postgres engine — as a
real provider with branches, credentials, and a Drizzle workflow
that wires straight into Cloudflare Hyperdrive. The second is a
focused pass on the **Layers** docs: the concept page now walks
through the encapsulation problem with an idiomatic init/runtime
split, and there's a step-by-step guide for building an
infrastructure Layer end-to-end.

## PlanetScale — MySQL + Postgres, planet-scale

`alchemy/Planetscale` is the new provider. It models the
PlanetScale control plane the same way the rest of alchemy models
clouds — top-level Database, Branch, credentials resource, and an
`origin` output that drops straight into Cloudflare Hyperdrive.
The two engines mirror each other:

| MySQL                       | Postgres                       |
| --------------------------- | ------------------------------ |
| `Planetscale.MySQLDatabase` | `Planetscale.PostgresDatabase` |
| `Planetscale.MySQLBranch`   | `Planetscale.PostgresBranch`   |
| `Planetscale.MySQLPassword` | `Planetscale.PostgresRole`     |

What follows is the development progression — every step adds one
piece on top of the previous.

### 1. Register the provider

```typescript
// alchemy.run.ts
export default Alchemy.Stack(
  "App",
  {
    providers: Cloudflare.providers(),
    state: Alchemy.localState(),
  },
  // ...
);
```

```diff lang="typescript"
+import * as Planetscale from "alchemy/Planetscale";
+import * as Layer from "effect/Layer";

 export default Alchemy.Stack(
   "App",
   {
-    providers: Cloudflare.providers(),
+    providers: Layer.mergeAll(
+      Cloudflare.providers(),
+      Planetscale.providers(),
+    ),
     state: Alchemy.localState(),
   },
   // ...
 );
```

`bun alchemy login` picks up the new step automatically — it
either reads `PLANETSCALE_API_TOKEN_ID` / `PLANETSCALE_API_TOKEN` /
`PLANETSCALE_ORGANIZATION` from the environment or stores them
under `~/.alchemy/credentials/<profile>/planetscale-stored.json`.

### 2. Declare the database

The long-lived cluster. Same shape for both engines — pick a
region and a `clusterSize`:

```typescript
// src/Db.ts
const database = yield* Planetscale.PostgresDatabase("app-db", {
  region: { slug: "us-east" },
  clusterSize: "PS_10",
});
```

```typescript
// src/Db.ts (MySQL variant)
const database = yield* Planetscale.MySQLDatabase("app-db", {
  region: { slug: "us-east" },
  clusterSize: "PS_10",
});
```

`region` and architecture are stable — changing them triggers a
replacement.

### 3. Branch off it

A branch is the cheap, per-feature unit you actually point your
Worker at. PR previews own ephemeral branches; long-lived stages
own the production branch:

```diff lang="typescript"
 const database = yield* Planetscale.PostgresDatabase("app-db", {
   region: { slug: "us-east" },
   clusterSize: "PS_10",
 });

+const branch = yield* Planetscale.PostgresBranch("app-branch", {
+  database,
+});
```

MySQL is the same call with one extra flag — `isProduction`
gates whether the branch can serve writes from the production
host:

```diff lang="typescript"
+const branch = yield* Planetscale.MySQLBranch("app-branch", {
+  database,
+  isProduction: false,
+});
```

### 4. Mint credentials

Postgres has `PostgresRole` (real Postgres roles with
`inheritedRoles`). MySQL has `MySQLPassword` (PlanetScale's
role-flavored password model):

```diff lang="typescript"
+const role = yield* Planetscale.PostgresRole("app-role", {
+  database,
+  branch,
+  inheritedRoles: ["postgres"],
+});
```

```diff lang="typescript"
+const password = yield* Planetscale.MySQLPassword("app-password", {
+  database,
+  branch,
+  role: "readwriter",
+});
```

Both expose an `origin: { scheme, host, port, database, user,
password }` output. The password is `Redacted` end-to-end — it
never lands in plan output or state files.

### 5. Front it with Hyperdrive

`origin` is the shape `Cloudflare.Hyperdrive` accepts directly, so
the wiring is one line:

```diff lang="typescript"
+const hd = yield* Cloudflare.Hyperdrive("app-hyperdrive", {
+  origin: role.origin, // or password.origin for MySQL
+});
```

Because `origin` is an `Output`, alchemy orders the deploy graph
for you: `Database` → `Branch` → `Role`/`Password` → `Hyperdrive`
→ `Worker`.

### 6. Drizzle, Postgres edition

The new `alchemy/Drizzle` module ships a Postgres helper that
returns a chainable Drizzle proxy from a Hyperdrive connection
string. It memoizes the pool against the Worker's isolated scope,
so you get one pool per isolate even though you call it from
every request:

```typescript
// src/Api.ts
import * as Cloudflare from "alchemy/Cloudflare";
import * as Drizzle from "alchemy/Drizzle";
import { relations, Users } from "./schema.ts";
import { Hyperdrive } from "./Db.ts";

export default class Api extends Cloudflare.Worker<Api>()(
  "Api",
  { main: import.meta.filename },
  Effect.gen(function* () {
    const conn = yield* Cloudflare.Hyperdrive.bind(Hyperdrive);
    const db = yield* Drizzle.postgres(conn.connectionString, {
      relations,
    });

    return {
      fetch: Effect.gen(function* () {
        const users = yield* db.select().from(Users);
        return yield* HttpServerResponse.json({ users });
      }),
    };
  }).pipe(Effect.provide(Cloudflare.HyperdriveBindingLive)),
) {}
```

`db.select().from(Users)` is a real Effect — no
`Effect.tryPromise` wrapper, no manual pool lifecycle. Errors land
in the Effect error channel.

### 7. Drizzle.Schema — migrations as a resource

`Drizzle.Schema` wraps drizzle-kit's programmatic API as an
alchemy resource. Pointing a `PostgresBranch.migrationsDir` at its
`out` output gives you a deploy-driven migration flow:

```diff lang="typescript"
+const schema = yield* Drizzle.Schema("app-schema", {
+  schema: "./src/schema.ts",
+  out: "./migrations",
+});

 const branch = yield* Planetscale.PostgresBranch("app-branch", {
   database,
+  migrationsDir: schema.out,
 });
```

The order falls out of the dependency graph:

1. `Drizzle.Schema` diffs the source against the latest stored
   snapshot and writes a new `{timestamp}_migration` directory
   when it drifts.
2. `Planetscale.PostgresBranch` scans the directory and applies
   any new SQL files transactionally.

### 8. MySQL — Drizzle over `mysql2`

MySQL pairs with `mysql2` and Drizzle's MySQL dialect. Migrations
are still resource-driven (`MySQLBranch.migrationsDir`), but the
runtime side stays closer to the raw driver:

```typescript
// src/Api.ts
import { drizzle } from "drizzle-orm/mysql2";
import * as Redacted from "effect/Redacted";

Effect.gen(function* () {
  const conn = yield* Cloudflare.Hyperdrive.bind(Hyperdrive);

  return {
    fetch: Effect.gen(function* () {
      const connectionString = yield* conn.connectionString;
      const db = drizzle(Redacted.value(connectionString), {
        schema,
        relations,
        mode: "default",
      });
      // ... db.select().from(Users), etc.
    }),
  };
});
```

End-to-end examples are checked in:

- [`examples/cloudflare-planetscale-postgres-drizzle`](https://github.com/alchemy-run/alchemy-effect/tree/main/examples/cloudflare-planetscale-postgres-drizzle)
- [`examples/cloudflare-planetscale-mysql-drizzle`](https://github.com/alchemy-run/alchemy-effect/tree/main/examples/cloudflare-planetscale-mysql-drizzle)

## Layers — refreshed concept doc + new guide

The Layers story got rewritten around how people actually reach
for Layers — when the leaky thing isn't a one-off `getJob`
helper, it's a Worker handler that's welded to one cloud's
binding.

The [Layers concept](/concepts/layers) now opens with a Worker
that calls KV directly:

```typescript
Effect.gen(function* () {
  const kv = yield* Cloudflare.KVNamespaceBinding.bind(MyKV);

  return {
    fetch: Effect.gen(function* () {
      const job = yield* kv.get<Job>("job-1", "json");
      return HttpServerResponse.json(job);
    }),
  };
});
```

— and walks through why that shape is the wrong abstraction
boundary (KV error, KV shape, KV binding all leaking into
`fetch`), then collapses the whole thing into a `JobService` Layer
that the consumer never has to know is KV-backed. The init /
runtime phase split now lines up with how `Cloudflare.Worker` is
actually written, instead of the ad-hoc `getJob(id)` example the
old version used. ([#386](https://github.com/alchemy-run/alchemy-effect/pull/386))

And there's a new step-by-step guide for putting that into
practice:

### New guide: Building Infrastructure Layers

[`/guides/infrastructure-layers`](/guides/infrastructure-layers)
is the "now build one yourself" companion to the concept doc. It
builds a `JobService` Layer backed by Cloudflare KV one delta at a
time — define the service, scaffold the Layer, declare the
resource, bind it, implement the methods, wire it into a Worker,
provide the runtime binding.

The progression is granular on purpose. One concept per heading,
one diff per step. Excerpted:

```typescript
// src/JobService.ts — define the contract
export class JobService extends Context.Service<
  JobService,
  {
    getJob(id: string): Effect.Effect<Job | null, never, Alchemy.RuntimeContext>;
    putJob(job: Job): Effect.Effect<void, never, Alchemy.RuntimeContext>;
  }
>()("JobService") {}
```

```diff lang="typescript"
 // src/JobService.KV.ts — add the resource
+import * as Cloudflare from "alchemy/Cloudflare";
 export const JobServiceKV = Layer.effect(
   JobService,
   Effect.gen(function* () {
+    const Namespace = yield* Cloudflare.KVNamespace("Jobs");
     return {
       // ...
     };
   }),
 );
```

```diff lang="typescript"
 // src/JobService.KV.ts — bind it
   Effect.gen(function* () {
     const Namespace = yield* Cloudflare.KVNamespace("Jobs");
+    const kv = yield* Cloudflare.KVNamespaceBinding.bind(Namespace);
     return {
       // ...
     };
   }),
```

```diff lang="typescript"
 // src/JobService.KV.ts — fill in the methods
     return {
       getJob: Effect.fn(function* (id: string) {
-        // TODO: read from KV
+        return yield* kv.get<Job>(id, "json");
       }),
       putJob: Effect.fn(function* (job: Job) {
-        // TODO: write to KV
+        yield* kv.put(job.id, JSON.stringify(job));
       }),
     };
```

```diff lang="typescript"
 // src/Api.ts — provide it
   Effect.gen(function* () {
+    const jobs = yield* JobService;
     return {
       fetch: Effect.gen(function* () {
+        const job = yield* jobs.getJob("job-1");
+        return yield* HttpServerResponse.json(job);
       }),
     };
-  }),
+  }).pipe(
+    Effect.provide(
+      JobServiceKV.pipe(Layer.provide(Cloudflare.KVNamespaceBindingLive)),
+    ),
+  ),
```

That last step is the punchline: `Layer.provide` satisfies
`JobServiceKV`'s dependency on `KVNamespaceBinding` *privately*,
so the consumer's required-context shrinks back to just
`JobService`. Swap the implementation for a DynamoDB-backed Layer,
the consumer's signature is unchanged. ([#383](https://github.com/alchemy-run/alchemy-effect/pull/383))
